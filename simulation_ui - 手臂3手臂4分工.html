瓶頸辨識使用說明

傳統「利用率最高 = 瓶頸」在連動式產線常常誤判 — 一台 100% 忙的設備可能是真瓶頸,也可能是被下游卡或被上游餓。模擬器把每個設備的時間細分為 5 種狀態。

1. 5 種狀態分類

█ P 產出真正在做有用工作
▓ B 擋想動但下游滿(出不去)
▒ S 餓想動但上游空(等不到料)
░ CG 週期等設計性等待(機台必要的循環)
· I 真閒沒明確原因的閒置

2. 三個關鍵 UI 區塊

  • 瓶頸設備卡:設備名 + 主因 + 5 分鐘最久連續產出
  • 各站狀態 5min 卡:全部設備按 P% 排名的 stacked bar
  • WIP 堆積卡:每個 buffer 的即時數量 + 5 分鐘平均

3. 判讀流程

  1. 跑滿 30 分鐘(讓暖機效應消失)
  2. 看「瓶頸設備」卡的主因:
    • 自身慢 + 最久連續 > 1 min → 真瓶頸,直接改該設備
    • 下游卡 ↓ → 真兇在下游
    • 上游缺 ↑ → 真兇在上游
    • 週期等 → cycle 結構問題
  3. WIP 黃金法則:
    「瓶頸的上游 WIP 滿、下游 WIP 空」。掃過 WIP 找「上面數字大、下面數字小」的交界。
  4. 用「最久連續」校驗:> 1 min = 真瓶頸 / < 10 sec = 假瓶頸
  5. 📊 匯出 → 改參數 → 重跑 → 再匯出 → diff

4. 常見陷阱

  • 看 P% 最高就以為是瓶頸 — 必須同時看主因和最久連續
  • 改了某設備但 UPH 沒變 → 改的是症狀,瓶頸已漂移
  • 多瓶頸接近 (P% 差 < 5%) → 產線平衡
  • WIP 全滿 → 瓶頸在最末端 (出料貨架/armB)
  • WIP 全空 → 瓶頸在最前端 (入料貨架/armA)
